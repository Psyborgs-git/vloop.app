## 2024-03-12 - [Terminal] Limit try-catch scope in input validation hot path
**Learning:** In V8 engines, wrapping a whole execution block (like looping over regex tests) inside a `try-catch` can prevent compiler optimizations, severely hindering performance on frequently called functions (hot paths) like payload input validation for terminals.
**Action:** When working on very hot paths where inputs stream at high frequency, narrow the scope of `try-catch` blocks specifically around the code that actually needs to catch expected exceptions (e.g. `new RegExp(pattern)` for invalid patterns), keeping the caching mechanism (`Map.get`) and fast-path execution (`re.test`) strictly outside the `try-catch` block.

## 2025-03-03 - [AI Agent Repos] Optimize recursive tree traversal with SQLite CTEs
**Learning:** Using a JavaScript `while` loop to iteratively fetch parent records one by one (e.g. `getAncestry` methods) introduces an N+1 query problem, severely impacting latency. Drizzle ORM does not yet have native built-in recursive fetching, but it supports raw SQL queries via the `sql` template literal. When bypassing the query builder with `sql`, the returned database rows retain their native snake_case field names instead of the mapped camelCase properties, which breaks type expectations.
**Action:** When retrieving hierarchical data like DAG paths or ancestries from SQLite, replace iterative ORM calls with a single `WITH RECURSIVE` Common Table Expression (CTE) executed via `drizzle-orm`'s `sql` utility. Remember to map the resulting raw snake_case rows using the corresponding `createRowMapper` module-level function before returning them to ensure consistency with TypeScript types.
