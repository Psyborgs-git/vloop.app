## 2024-03-12 - [Terminal] Limit try-catch scope in input validation hot path
**Learning:** In V8 engines, wrapping a whole execution block (like looping over regex tests) inside a `try-catch` can prevent compiler optimizations, severely hindering performance on frequently called functions (hot paths) like payload input validation for terminals.
**Action:** When working on very hot paths where inputs stream at high frequency, narrow the scope of `try-catch` blocks specifically around the code that actually needs to catch expected exceptions (e.g. `new RegExp(pattern)` for invalid patterns), keeping the caching mechanism (`Map.get`) and fast-path execution (`re.test`) strictly outside the `try-catch` block.

## 2024-03-18 - [Optimize DAG Traversal with Recursive CTEs]
**Learning:** [In a DAG-based data structure stored in SQLite (like `StateNodeRepo` and `MessageRepo` within `packages/ai-agent`), resolving ancestry using recursive N+1 `.get()` lookups is an anti-pattern that degrades performance as execution chains grow long. Drizzle ORM does not provide a native `withRecursive` feature out of the box for Better SQLite3, but raw `sql` template literals can efficiently replace loops with a single round-trip recursive CTE query, bypassing Drizzle's ORM row mapping and requiring manual mapping (e.g. `mapStateNode` / `mapMessage`) and an array `.reverse()` to preserve chronological root-to-leaf order.]
**Action:** [When implementing DAG traversals in similar environments, default to raw SQL recursive CTEs using `this.orm.all(query) as Record<string, unknown>[]` and apply module-level mapper functions to avoid N+1 DB lookup overhead.]
