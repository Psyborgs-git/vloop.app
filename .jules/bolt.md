## 2024-03-12 - [Terminal] Limit try-catch scope in input validation hot path
**Learning:** In V8 engines, wrapping a whole execution block (like looping over regex tests) inside a `try-catch` can prevent compiler optimizations, severely hindering performance on frequently called functions (hot paths) like payload input validation for terminals.
**Action:** When working on very hot paths where inputs stream at high frequency, narrow the scope of `try-catch` blocks specifically around the code that actually needs to catch expected exceptions (e.g. `new RegExp(pattern)` for invalid patterns), keeping the caching mechanism (`Map.get`) and fast-path execution (`re.test`) strictly outside the `try-catch` block.

## 2024-03-22 - [AI Agent] Use Recursive CTEs for DAG traversal in Drizzle ORM
**Learning:** In Drizzle ORM, iteratively calling `this.get(parentId)` to traverse a DAG like `MessageRepo` or `StateNodeRepo` ancestry causes an N+1 query bottleneck. While `db.$with(...)` is native for CTEs, raw SQL `WITH RECURSIVE` queries using Drizzle's `sql` literal (e.g., `this.orm.all(sql\`. . .\`)`) execute in a single round-trip and provide massive speedups (e.g., ~11x faster for a 20-node chain). However, because `this.orm.all()` returns raw `snake_case` SQLite rows, these rows must be carefully mapped back to `camelCase` TypeScript objects before returning. In this repo, we can safely reuse the `createRowMapper` output (e.g., `mapStateNode`) to handle this conversion natively.
**Action:** Always prefer Recursive CTEs in Drizzle for traversing DAGs or tree structures. When using raw SQL via `this.orm.all()`, remember to explicitly apply the appropriate row mappers to bridge the `snake_case` database rows back to domain `camelCase` objects.
